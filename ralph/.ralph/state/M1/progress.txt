## Codebase Patterns
- Verify command: CLI args defined inline in `Commands::Verify` enum in `main.rs`, matched in `main()` to call `commands::verify::run()`
- Verify resolution: `verifier` arg is `&str` (now `&[String]`), validated against hardcoded list in `run()` before any logic
- Both `run()` and `run_batch()` use `VerifyOpts` from `treb-verify` crate to build forge verification args
- Registry update_deployment() called ONCE after all verifiers complete per deployment (not per-verifier)
- Aggregate status: all VERIFIED -> Verified, all FAILED -> Failed, mixed -> Partial (via `aggregate_status()` helper)
- Test file `cli_verify.rs` has unit-level CLI tests; `integration_verify.rs` has integration tests needing a seeded registry
- Per-verifier config resolution: `resolve_api_key(verifier, &explicit_key)` checks env vars (ETHERSCAN_API_KEY, BLOCKSCOUT_API_KEY) when no explicit key provided
- Styled output pattern: `fn styled(text, style) -> String` checks `color::is_color_enabled()`, uses `text.style(style)` from owo_colors
- Stage indicators: `output::print_stage(emoji, message)` prints to stderr with STAGE color styling
- Palette constants: `color::VERIFIED` (green bold), `color::FAILED` (red bold), `color::MUTED` (dimmed) for verify status display
