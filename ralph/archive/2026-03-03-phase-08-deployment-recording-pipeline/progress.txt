## Codebase Patterns
- Pipeline module lives at `crates/treb-forge/src/pipeline/` as a submodule directory with `mod.rs` + `types.rs`
- New modules: create file, register in `lib.rs` with `pub mod`, add re-exports with `pub use`
- Core domain types live in `treb-core/src/types/` (Deployment, Transaction, SafeTransaction)
- Registry CRUD via `treb-registry::Registry` — `insert_deployment()`, `get_deployment()`, etc.
- Alloy primitives (`Address`, `B256`, `Bytes`) are the forge-domain types; core domain uses `String` for addresses/hashes
- `Address::to_checksum(None)` converts Address to checksummed hex string
- `format!("{:#x}", b256_value)` gives 0x-prefixed lowercase hex for B256
- `ExtractedDeployment` (forge domain) → `Deployment` (core domain) is the hydration step
- `treb-registry` and `treb-config` are both workspace dependencies available to treb-forge
- `DeploymentMethod` enum is in `treb_core::types::enums` (Create, Create2, Create3)
- Tests use `#[cfg(test)] mod tests` inline pattern, not separate test files for unit tests
- `hydrate_deployment()` converts ExtractedDeployment → Deployment; `generate_deployment_id()` builds the ID string
- B256::ZERO → empty string; non-zero B256 → `format!("{:#x}", value)` for 0x-prefixed hex
- `Bytes` display gives `0x`-prefixed hex; use `.is_empty()` check for empty-string fallback
- `foundry_compilers::ArtifactId` requires `profile` field and `foundry_config::semver::Version` (not bare `semver`)
- ProxyType (forge) → String (core): "Transparent", "UUPS", "Beacon", "Minimal"
- `chrono` workspace dep needed in treb-forge for `Utc::now()` timestamps
- `TransactionSimulated` event contains `Vec<SimulatedTransaction>`; each SimulatedTransaction becomes one core `Transaction`
- `SafeTransactionQueued` event has indexed fields (`safeTxHash`, `safe`, `proposer`) + non-indexed `transactionIds: Vec<B256>`
- Transaction-to-deployment linking: match `Deployment.transaction_id` (hex string) against SimulatedTransaction.transactionId (B256 → hex)
- `hydrate_transactions()` and `hydrate_safe_transactions()` live alongside `hydrate_deployment()` in `pipeline/hydration.rs`
- Duplicate detection: `check_duplicate()` checks by ID first (SameId), then by address+chain_id (SameAddress); `resolve_duplicates()` applies Skip/Error/Update strategy
- `Registry::list_deployments()` returns `Vec<&Deployment>` sorted by created_at — use for address+chain scans
- `tempfile` crate is available as dev-dependency in treb-forge for tests needing `Registry::init()`
- Add `#[derive(Debug)]` to pipeline result types (e.g. `ResolvedDuplicates`, `SkippedDeployment`) if tests need `.unwrap_err()` on `Result<T, _>`
- `RunPipeline` orchestrator lives in `pipeline/orchestrator.rs`; the flow is: compile → execute → decode → extract → hydrate → deduplicate → record
- `Config::load_with_root()` returns `Result<Config, ExtractConfigError>`, not `Config` directly — must handle the error
- sol! event types (TransactionSimulated, SafeTransactionQueued) derive Clone via alloy — safe to `.clone()` when extracting from ParsedEvent
- `foundry_config::Config` is loaded via `Config::load_with_root(project_root).map(|c| c.sanitized())` pattern
- All core domain types (Deployment, Transaction, SafeTransaction) derive `Clone` — can clone before registry insert to keep a copy for PipelineResult
