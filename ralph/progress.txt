## Codebase Patterns
- foundry-wallets crate is at `crates/wallets` inside the foundry git repo, pinned to tag v1.5.1
- WalletSigner::from_private_key() takes &B256 (not a hex string) — parse hex first with `alloy_primitives::hex::FromHex`
- WalletSigner::from_ledger_path() and from_trezor_path() are async (require hardware wallet connection)
- WalletSigner::from_mnemonic() takes mnemonic string, optional passphrase, optional derivation path, and index
- LedgerHDPath and TrezorHDPath are from alloy_signer_ledger and alloy_signer_trezor respectively (re-exported through foundry-wallets signer.rs)
- Anvil's default mnemonic: "test test test test test test test test test test test junk" — used for in-memory signers
- treb-forge module pattern: module file + declaration in lib.rs + `pub use` re-exports of key types/functions
- ScriptArgs.wallets is MultiWalletOpts (CLI arg struct) — inject private keys via `wallets.private_key = Some(hex_string)`
- ResolvedSender.sender_address() returns the on-chain address for any variant (Wallet/InMemory → signer address, Safe → safe address, Governor → governor address)
- build_script_config_with_senders() extends build_script_config() with wallet integration — takes &HashMap<String, ResolvedSender>
- All treb-forge public functions return `treb_core::Result<T>` (TrebError)
- Config validation errors use TrebError::Config(), forge/wallet API errors use TrebError::Forge()
- alloy_primitives::hex::FromHex::from_hex() handles both 0x-prefixed and bare hex strings
- Import `alloy_signer::Signer` trait to call `.address()` on WalletSigner
- LedgerHDPath = alloy_signer_ledger::HDPath; TrezorHDPath = alloy_signer_trezor::HDPath
- Recursive async functions in Rust require `Pin<Box<dyn Future>>` return type instead of `async fn` — use `Box::pin(async move { ... })`
- Safe resolution: uses `config.safe` (or `config.address` fallback) for safe address, `config.signer` for sub-sender name
- Governor resolution: uses `config.governor` (or `config.address` fallback), optional `config.timelock`, `config.proposer` for sub-sender name
- alloy sol! macro: use `alloy_sol_types::sol;` — generates typed structs with `SIGNATURE_HASH` (via `SolEvent` trait), `decode_log_data()`, etc.
- alloy sol! macro: add `#[derive(Debug)]` inside sol! blocks on each struct/event to enable Debug formatting on generated types
- SolEvent::decode_log_data(&log.data) takes 1 arg (no validate bool) in alloy-sol-types 1.5.x
- LogData fields: `log.data.topics()` returns `&[B256]`, `log.data.data` is the raw bytes (field, not method)
- Log::new(address, topics_vec, data_bytes) constructs a Log from parts; returns Option
- alloy sol! macro overloaded events: disambiguated as `EventName_0`, `EventName_1`, etc. (e.g., `ContractCreation_0` for with-salt variant)
- alloy sol! macro clippy: don't put doc comments followed by blank lines inside sol! blocks — triggers `empty_line_after_outer_attr` warning
- Import `alloy_sol_types::SolEvent` trait to access `SIGNATURE_HASH`, `decode_log()`, etc. on sol!-generated types
